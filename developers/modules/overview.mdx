## Module Integration Pattern

Most Neutron modules follow a consistent integration pattern when used in CosmWasm smart contracts:

1. **Import the Dependencies**
   ```rust
   use neutron_sdk::interchain_queries::...; // for the ICQ module
   use neutron_sdk::bindings::...; // for other modules
   ```

2. **Create Module-Specific Messages**
   ```rust
   // Example for registering an interchain query
   let msg = NeutronMsg::RegisterInterchainQuery { ... };
   ```

3. **Handle Response/Callbacks**
   ```rust
   // Most modules provide callbacks via Sudo messages
   #[entry_point]
   pub fn sudo(deps: DepsMut, env: Env, msg: SudoMsg) -> StdResult<Response> {
       match msg {
           SudoMsg::ModuleSpecificCallback { ... } => { ... }
           _ => Err(StdError::generic_err("unsupported sudo message")),
       }
   }
   ```

## Choosing the Right Modules

When building on Neutron, consider which modules best suit your application's needs:

- **Cross-Chain Data Needs**: Use Interchain Queries (ICQ) to access data from other chains
- **Cross-Chain Transaction Needs**: Use Interchain Transactions for executing operations on remote chains
- **Automation Requirements**: Use Cron for scheduled operations
- **Price Feed Requirements**: Use Oracle for reliable price data
- **Fee Management**: Use Fee Burner and Fee Refunder for fee economics and IBC relayer incentives
- **Smart Contract Safety**: Use Contract Manager to handle IBC failures gracefully
- **Exchange Functionality**: Use Dex for token swaps and liquidity provision
- **Revenue Capture**: Use Revenue module to earn from your dApp usage
- **Asset Management**: Use Harpoon for delegated asset control
- **Token Transfer Needs**: Use Transfer for cross-chain asset movement
- **Network Protection**: Use IBC Rate Limit to safeguard against excessive outflows
